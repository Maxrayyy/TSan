> 版本: v1.0 | 日期: 2026-04-07

## 1. 技术选型

### 1.1 总体架构

```
┌─────────────────────────────────────────────────────┐
│                    客户端 (Browser)                    │
│    React + TypeScript + Socket.IO Client + Zustand    │
└──────────────────────┬──────────────────────────────┘
                       │ HTTPS / WSS
                       ▼
┌─────────────────────────────────────────────────────┐
│                   Nginx (反向代理/静态资源)             │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│              Node.js 应用服务器                        │
│         Express + Socket.IO + TypeScript              │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────┐   │
│  │ REST API │  │ WS 网关   │  │ 游戏引擎(核心逻辑) │   │
│  └────┬─────┘  └────┬─────┘  └────────┬─────────┘   │
│       │             │                  │              │
└───────┼─────────────┼──────────────────┼─────────────┘
        │             │                  │
   ┌────▼─────┐  ┌───▼────┐   ┌────────▼────────┐
   │PostgreSQL│  │ Redis  │   │  Redis (Pub/Sub) │
   │(持久数据) │  │(游戏状态)│   │  (多实例通信)     │
   └──────────┘  └────────┘   └──────────────────┘
```

### 1.2 技术栈详情

| 层级 | 技术 | 选型理由 |
|------|------|----------|
| **前端框架** | React 18 + TypeScript | 生态成熟，组件化开发，类型安全 |
| **前端状态** | Zustand | 轻量，适合游戏状态管理，比Redux简洁 |
| **前端构建** | Vite | 开发体验好，HMR快 |
| **前端UI** | TailwindCSS + Headless UI | 原子化CSS，快速迭代，响应式支持好 |
| **前端通信** | Socket.IO Client | 自动重连、房间支持、回退机制 |
| **后端运行时** | Node.js 20 LTS + TypeScript | 全栈统一语言，异步IO适合实时应用 |
| **后端框架** | Express.js | 轻量，仅处理REST API；WebSocket由Socket.IO处理 |
| **实时通信** | Socket.IO | 房间抽象、自动重连、二进制支持 |
| **主数据库** | PostgreSQL 16 | 关系型，事务支持好，存用户/战绩 |
| **缓存/状态** | Redis 7 | 游戏房间状态、会话、Pub/Sub |
| **ORM** | Prisma | 类型安全，迁移管理方便 |
| **认证** | JWT (access + refresh token) | 无状态，支持游客模式 |
| **部署** | Docker + Docker Compose | 本地开发一致性，易于部署 |
| **反向代理** | Nginx | 静态资源、SSL终端、WebSocket代理 |

### 1.3 为什么不用其他方案

| 备选 | 不选原因 |
|------|----------|
| Next.js | 牌类游戏不需要SSR/SSG，纯SPA更合适 |
| WebSocket原生 | Socket.IO提供房间、重连、广播等开箱即用功能 |
| MongoDB | 游戏数据关系明确（用户-房间-对局），关系型更合适 |
| Go/Rust后端 | 初期开发效率优先，前后端统一TypeScript降低维护成本 |

## 3. 通信协议设计

### 3.1 REST API（非实时操作）

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/auth/register` | 注册 |
| POST | `/api/auth/login` | 登录 |
| POST | `/api/auth/guest` | 游客登录（仅传昵称） |
| GET | `/api/user/profile` | 获取用户信息 |
| PUT | `/api/user/profile` | 更新用户信息 |
| GET | `/api/user/stats` | 获取战绩统计 |
| POST | `/api/room/create` | 创建房间 |
| GET | `/api/room/:id` | 查询房间信息 |

### 3.2 Socket.IO 事件（实时操作）

#### 客户端 → 服务端

| 事件名 | 数据 | 说明 |
|--------|------|------|
| `room:join` | `{ roomId, seatIndex? }` | 加入房间 |
| `room:leave` | `{}` | 离开房间 |
| `room:ready` | `{}` | 准备 |
| `room:start` | `{}` | 房主开始游戏 |
| `room:chat` | `{ message }` | 发送聊天消息 |
| `game:play` | `{ cards: Card[] }` | 出牌 |
| `game:pass` | `{}` | 不要（pass） |
| `game:declare` | `{ type }` | 扑牌声明 |

#### 服务端 → 客户端

| 事件名 | 数据 | 说明 |
|--------|------|------|
| `room:state` | `RoomState` | 房间完整状态（加入/重连时） |
| `room:player-joined` | `{ player, seatIndex }` | 有人加入 |
| `room:player-left` | `{ playerId }` | 有人离开 |
| `room:player-ready` | `{ playerId }` | 有人准备 |
| `room:chat` | `{ playerId, message, timestamp }` | 聊天消息 |
| `game:start` | `{ hands, firstPlayer, gameState }` | 游戏开始（各自手牌） |
| `game:your-turn` | `{ timeLimit, validActions }` | 轮到你 |
| `game:played` | `{ playerId, cards, handType }` | 某人出牌 |
| `game:passed` | `{ playerId }` | 某人pass |
| `game:round-end` | `{ winnerId, score, tuoSan? }` | 回合结束 |
| `game:tuo-san` | `{ playerId, count }` | 拖三触发 |
| `game:bie-san` | `{ playerId, count }` | 憋三触发 |
| `game:player-finished` | `{ playerId, rank }` | 某人出完牌 |
| `game:end` | `{ result: GameResult }` | 游戏结束 |
| `game:reconnect` | `{ fullGameState }` | 断线重连恢复 |
| `error` | `{ code, message }` | 错误信息 |

## 4. 状态管理

### 4.1 服务端游戏状态（存Redis）

```typescript
interface GameState {
  id: string;                    // 游戏ID
  roomId: string;                // 房间ID
  phase: 'dealing' | 'declaring' | 'playing' | 'finished';
  
  players: {                     // 4个玩家
    [seatIndex: number]: {
      userId: string;
      hand: Card[];              // 手中的牌
      score: number;             // 本局已抓分
      rank: number | null;       // 排名（null=未出完）
      tuoSanCount: number;       // 拖三数
      bieSanCount: number;       // 憋三数
      connected: boolean;        // 连接状态
    };
  };
  
  currentPlayerSeat: number;     // 当前出牌者座位
  turnStartTime: number;         // 本次出牌开始时间
  
  currentRound: {                // 当前回合信息
    leadPlayerSeat: number;      // 本回合首出者
    lastPlay: {                  // 场上当前最大的牌
      playerSeat: number;
      cards: Card[];
      handType: HandType;
    } | null;
    passCount: number;           // 连续pass次数
    roundScore: number;          // 本回合场上分数牌总分
  };
  
  teamScores: [number, number];  // [队伍A总分, 队伍B总分]
  finishedOrder: number[];       // 出完牌的顺序
  isFirstPlay: boolean;          // 是否为本局首次出牌
  roundHistory: RoundRecord[];   // 历史回合记录
}
```

## 5. 安全架构

### 5.1 服务端权威原则

**核心原则**：所有游戏逻辑判定均在服务端完成，客户端仅为展示层。

- 客户端发送 `game:play({ cards })` → 服务端验证合法性 → 合法则广播，非法则返回错误
- 客户端永远不知道其他玩家的手牌
- 洗牌、发牌在服务端执行
- 计分在服务端完成

### 5.2 认证与授权

```
┌──────────┐    POST /auth/login     ┌──────────┐
│  Client  │ ──────────────────────> │  Server  │
│          │ <────────────────────── │          │
│          │    { accessToken,       │          │
│          │      refreshToken }     │          │
│          │                         │          │
│          │  Socket.IO handshake    │          │
│          │  auth: { token }        │          │
│          │ ──────────────────────> │          │
│          │  (验证JWT, 绑定userId)   │          │
└──────────┘                         └──────────┘
```

- Access Token: 有效期2小时，短期
- Refresh Token: 有效期7天，用于刷新access token
- 游客Token: 服务端生成临时userId，Access Token有效期24小时
- Socket连接时通过auth传递token，服务端中间件验证

### 5.3 防作弊

- 发送到客户端的游戏状态经过过滤，不包含其他玩家手牌
- 出牌验证：检查玩家确实持有这些牌
- 时序验证：只有当前轮到的玩家可以出牌
- 频率限制：防止恶意刷请求

## 6. 部署架构

### 6.1 开发环境

```yaml
# docker-compose.yml
services:
  postgres:
    image: postgres:16-alpine
    ports: ["5432:5432"]
    volumes: [pgdata:/var/lib/postgresql/data]
    
  redis:
    image: redis:7-alpine
    ports: ["6379:6379"]
    
  server:
    build: ./packages/server
    ports: ["3000:3000"]
    depends_on: [postgres, redis]
    
  client:
    build: ./packages/client
    ports: ["5173:5173"]
```

### 6.2 生产环境（推荐）

```
                    ┌─── Cloudflare CDN ───┐
                    │   (静态资源 + SSL)      │
                    └──────────┬────────────┘
                               │
                    ┌──────────▼────────────┐
                    │     Nginx / Caddy      │
                    │  (反向代理 + 负载均衡)   │
                    └──────────┬────────────┘
                               │
              ┌────────────────┼────────────────┐
              ▼                ▼                 ▼
        ┌──────────┐   ┌──────────┐     ┌──────────┐
        │ Node.js  │   │ Node.js  │     │ Node.js  │
        │ 实例 1    │   │ 实例 2    │     │ 实例 N    │
        └────┬─────┘   └────┬─────┘     └────┬─────┘
             │              │                 │
             └──────────────┼─────────────────┘
                            │
              ┌─────────────┼─────────────┐
              ▼                           ▼
        ┌──────────┐              ┌──────────┐
        │PostgreSQL│              │  Redis   │
        │ (主从)    │              │ (Pub/Sub)│
        └──────────┘              └──────────┘
```

- Socket.IO 使用 Redis Adapter 支持多实例
- 房间状态存 Redis，多实例可共享
- 静态资源走CDN

### 6.3 最小部署方案（单机）

适合初期：1台 2核4G 云服务器即可
- Nginx + Node.js + PostgreSQL + Redis 全部跑在一台机器
- 使用 Docker Compose 编排
- 预估可支撑 200 个并发房间

## 7. 关键技术决策记录

| # | 决策 | 理由 |
|---|------|------|
| 1 | Monorepo + pnpm workspace | 前后端共享类型定义，开发体验好 |
| 2 | 游戏状态存 Redis 而非 PostgreSQL | 游戏状态读写频繁、生命周期短，Redis性能更优 |
| 3 | 游戏结果（对局记录）持久化到 PostgreSQL | 需要长期存储、查询统计 |
| 4 | 不使用游戏引擎(Phaser等) | 扑克牌游戏UI简单，CSS动画足够，不需要Canvas |
